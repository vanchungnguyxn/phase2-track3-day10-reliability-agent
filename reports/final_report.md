# Day 10 Reliability Report

## 1. Architecture summary

Gateway nhận prompt từ client, kiểm tra **shared Redis cache** trước (semantic similarity + privacy guardrails). Cache miss thì đi qua **circuit breaker** của từng provider theo thứ tự primary → backup. Circuit OPEN thì fail-fast, không retry storm. Tất cả provider thất bại thì trả **static fallback**.

```
User Request
    |
    v
[ReliabilityGateway]
    |
    +---> [SharedRedisCache.get()] -- HIT? --> return (route=cache_hit:score, cost=0)
    |              |
    |              v MISS
    +---> [CircuitBreaker: primary] --> FakeLLMProvider(primary)
    |         | OPEN / ProviderError
    |         v
    +---> [CircuitBreaker: backup]  --> FakeLLMProvider(backup)
    |         | all failed
    |         v
    +---> [Static fallback message]
```

**Thành phần chính:**

| Layer | File | Vai trò |
|---|---|---|
| Gateway | `gateway.py` | Orchestrate cache → breaker → fallback |
| Circuit breaker | `circuit_breaker.py` | CLOSED / OPEN / HALF_OPEN, transition log |
| Cache | `cache.py` | `ResponseCache` (in-memory) + `SharedRedisCache` (Redis HSET/SCAN) |
| Chaos | `chaos.py` | 3 scenarios, concurrent load, pass/fail criteria, cache A/B |
| Metrics | `metrics.py` | P50/P95/P99, availability, JSON + CSV export |

Load test chạy với **4 concurrent workers** (`ThreadPoolExecutor`) để mô phỏng traffic song song và kích hoạt circuit breaker nhanh hơn.

## 6. Redis shared cache

### Tại sao cần shared cache?

- **In-memory cache** chỉ tồn tại trong process một gateway instance. Khi scale horizontal (3+ pods), mỗi instance có cache riêng → hit rate thấp, cost tăng, latency không đồng nhất.
- **`SharedRedisCache`** lưu `query` + `response` trong Redis Hash (`HSET`), TTL qua `EXPIRE`, key `rl:cache:{md5_hash}`. Mọi instance đọc cùng namespace → cache warming một lần, tất cả instance hưởng lợi.

## 8. Failure analysis

### Điểm yếu còn lại

**Recovery time không đo được trong một số scenario concurrent.** Khi chạy 4 workers song song, circuit có thể OPEN nhưng chưa kịp probe HALF_OPEN → CLOSED trước khi scenario kết thúc (100 requests ~18s). `recovery_time_ms` aggregate = `null` dù `without_cache` run đo được ~2253 ms.

### Đề xuất sửa trước production

1. **Thread-safe circuit breaker** — counters và state transitions cần lock khi dùng concurrent workers; hiện tại race condition có thể đếm sai `failure_count`.
2. **Redis circuit state** — chia sẻ breaker state qua Redis INCR/EXPIRE để multi-instance không mỗi nơi một circuit riêng.
3. **SCAN → index** — similarity lookup qua `SCAN` là O(keys); production nên dùng vector DB hoặc Redis Search module.
